# TON Whale Watcher

Telegram-бот, который следит за крупными покупками (свопами) на DEX в блокчейне TON
и присылает алерт, если сумма сделки превышает заданный порог в USD (по умолчанию $1000),
а также отслеживает аномальные всплески объёма торгов по отдельным токенам.

## Зачем этот проект

Цель — потренироваться строить сквозной пайплайн для потоковых данных: от подключения
к внешнему API до формулировки критерия «аномалии» и доставки результата пользователю
в удобном виде. Это тот же навык, что и в работе с клиентскими обращениями (например,
в CRM), только источник данных здесь — блокчейн.

Практический цикл, который отрабатывался:

1. собрать сырые данные (события DEX через TonAPI);
2. агрегировать и посчитать метрики (pandas: объём за окно, скользящее сравнение);
3. сформулировать и проверить критерий «аномалии» (во сколько раз должен вырасти объём,
   чтобы это было сигналом, а не шумом);
4. упаковать результат в понятный пользователю формат (сообщение в Telegram).

Роль «заказчика» в этом пайплайне я выполнял сам для себя.

## Что реализовано

- Скрипт на Python опрашивает индексатор блокчейна TON ([TonAPI](https://tonapi.io))
  и получает своп-операции на DEX в реальном времени.
- Суммы сделок переводятся в USD по текущему курсу TON и сравниваются с порогом;
  при превышении — алерт в Telegram.
- Отдельный модуль на pandas (`volume_tracker.py`) отслеживает объём торгов по каждому
  токену: сравнивает объём за последние 5 минут с обычным объёмом за такое же окно
  (на основе истории за последний час) и сигнализирует, если объём вырос в 3+ раза —
  часто это первый признак, что в токен начали заходить деньги.
- Защита от дублей алертов (по `lt` — logical time последнего обработанного события)
  и от спама одинаковыми сигналами по объёму (cooldown).
- Сделки и алерты сохраняются в SQLite (`db.py`) для последующей аналитики
  (топ токенов по объёму, статистика по спайкам) — без внешних зависимостей,
  только стандартная библиотека `sqlite3`.
- Логика детекции спайков покрыта тестами на синтетических данных (`tests/`) —
  это позволило поймать ошибки в пороговых значениях ещё до подключения к живому API.

## Результат

Рабочий прототип с двумя типами алертов — на разовые крупные сделки и на аномальные
всплески объёма — плюс минимальный слой хранения данных для аналитики поверх реальных
событий блокчейна.

## Как это устроено

1. **Источник данных** — [TonAPI](https://tonapi.io), публичный индексатор блокчейна TON.
   Используется эндпоинт `/v2/accounts/{id}/events`, который возвращает уже
   распарсенные события аккаунта, включая своп-операции на DEX (STON.fi, DeDust и др.).
2. **Опрос по расписанию** — раз в `POLL_INTERVAL` секунд скрипт запрашивает новые
   события для списка "слушаемых" адресов (роутеры/пулы DEX). Важно: запрос идёт
   **не по временному окну** ("события за последние N минут"), а по курсору `after_lt`
   (logical time) — API отдаёт до 50 событий, произошедших **после** позиции, на
   которой бот остановился в прошлый раз. Это надёжнее, чем опрос по времени: не
   зависит от рассинхронизации часов и не теряет события при коротких простоях.
3. **Фильтрация** — из всех событий выбираются действия типа `JettonSwap`,
   сумма переводится в USD по текущему курсу TON (`/v2/rates`) и сравнивается с порогом.
4. **Алерт** — если сумма выше порога, бот отправляет сообщение в Telegram через Bot API.
5. **Дедупликация** — скрипт хранит `lt` последнего обработанного события в
   `state.json`, чтобы не слать один и тот же алерт дважды и продолжать с того же
   места после перезапуска.

> Отдельно от этого механизма — «окно в 5 минут» из `volume_tracker.py`
> (`RECENT_WINDOW_MIN`) не имеет отношения к запросам к TonAPI. Это чисто локальный
> анализ: бот копит в памяти все полученные через API сделки с их временными метками
> и уже сам, без дополнительных запросов, сравнивает объём за последние 5 минут с
> обычным объёмом за час — на каждой итерации цикла.

## Установка и запуск с нуля

### 1. Клонируйте репозиторий и создайте виртуальное окружение

```bash
git clone https://github.com/<ваш-профиль>/ton-whale-watcher.git
cd ton-whale-watcher
python3 -m venv venv
source venv/bin/activate      # на Windows: venv\Scripts\activate
```

Нужен Python 3.10+.

### 2. Установите зависимости

```bash
pip install -r requirements.txt
```

### 3. Создайте бота в Telegram

1. В Telegram напишите [@BotFather](https://t.me/BotFather) → `/newbot` → задайте имя
   и username (должен заканчиваться на `bot`) → получите токен.
2. Никогда не публикуйте этот токен (в коде, коммитах, скриншотах) — это полный
   доступ к боту.

### 4. Настройте переменные окружения

```bash
cp .env.example .env
```

Откройте `.env` и впишите токен:
```
TELEGRAM_BOT_TOKEN=токен_от_BotFather
```

Файлы читают переменные из окружения процесса, поэтому перед запуском их также
нужно экспортировать в терминал:
```bash
export TELEGRAM_BOT_TOKEN="токен_от_BotFather"
```

### 5. Узнайте chat_id

1. Откройте чат с вашим ботом в Telegram и отправьте ему `/start` (или любое
   сообщение). Если хотите слать алерты в отдельный канал/группу — добавьте туда
   бота и напишите там.
2. Сразу после этого откройте в браузере (подставив свой токен):
   ```
   https://api.telegram.org/bot<ВАШ_ТОКЕН>/getUpdates
   ```
3. В ответе найдите `"chat":{"id": ЧИСЛО, ...}` — это и есть `TELEGRAM_CHAT_ID`
   (для каналов/групп он обычно отрицательный). Если `result` пустой — значит
   Telegram не увидел сообщение; отправьте его ещё раз и сразу обновите страницу.

Впишите значение в `.env` и экспортируйте:
```bash
export TELEGRAM_CHAT_ID="полученный_id"
```

### 6. Проверьте связь с Telegram

```bash
python tests/test_telegram_connection.py
```

Если в чат пришло тестовое сообщение — связка настроена верно.

### 7. Получите ключ TonAPI (настоятельно рекомендуется)

Без ключа TonAPI даёт очень скромный лимит запросов, и опрос быстро упирается
в `429 Too Many Requests`. Зарегистрируйтесь на [tonconsole.com](https://tonconsole.com),
создайте проект и сгенерируйте бесплатный API-ключ:

```bash
export TONAPI_KEY="ваш_ключ"
```
(и впишите в `.env` для постоянного использования).

Если ключа пока нет — можно временно увеличить `POLL_INTERVAL` (например до 60),
чтобы реже дёргать API, но для стабильной работы ключ лучше получить сразу.

### 8. Запустите бота

```bash
export THRESHOLD_USD=1000
python watcher.py
```

Все переменные окружения:

| Переменная            | Обязательна | Описание                                              |
|-----------------------|-------------|--------------------------------------------------------|
| `TELEGRAM_BOT_TOKEN`  | да          | токен бота от @BotFather                               |
| `TELEGRAM_CHAT_ID`    | да          | ID чата/канала, куда слать алерты                       |
| `TONAPI_KEY`          | нет*        | ключ TonAPI (бесплатный, tonconsole.com) — без него быстро упрётесь в 429 |
| `WATCH_ACCOUNTS`      | нет         | адреса через запятую (роутеры/пулы DEX)                 |
| `THRESHOLD_USD`       | нет         | порог алерта в USD (по умолчанию 1000)                  |
| `POLL_INTERVAL`       | нет         | интервал опроса в секундах (по умолчанию 20)            |

## Важно: первый запуск

При первом обнаруженном свопе скрипт напечатает в консоль "сырой" JSON
действия `JettonSwap`. Сверьтесь с ним: если поля называются иначе, чем
`ton_in`/`ton_out`/`dex` (структура TonAPI может немного меняться между
версиями), поправьте функцию `parse_swap_action()` в `watcher.py` —
там всего несколько строк.

## Тесты

**Автоматические** — логика детекции всплесков объёма проверена на синтетических
данных, без обращения к живому API:

```bash
python -m pytest tests/test_volume_tracker.py -v
# или
python tests/test_volume_tracker.py
```

Проверяются: отсутствие ложных алертов на ровном объёме, срабатывание на резком
росте, работа cooldown против дублей и защита от лже-алертов на почти нулевой истории.

**Ручной** — `tests/test_telegram_connection.py` реально обращается к Telegram API
и шлёт тестовое сообщение в чат. Используется один раз при настройке, чтобы
убедиться, что токен и chat_id верны (см. шаг 6 в разделе установки):

```bash
python tests/test_telegram_connection.py
```

## Хранение данных (модуль `db.py`)

Все сделки и алерты о всплесках объёма сохраняются в локальную базу SQLite
(`trades.db`, создаётся автоматически). Используется только стандартная
библиотека Python (`sqlite3`) — без внешних зависимостей.

Простые аналитические запросы уже есть в модуле:

```python
import db

# топ токенов по объёму за последние 24 часа
db.top_tokens_by_volume(hours=24, limit=10)

# сколько алертов о всплеске объёма было по каждому токену за неделю
db.spikes_count_last_days(days=7)
```

Под капотом — обычные `SELECT ... GROUP BY ... ORDER BY ... LIMIT`
с фильтрацией по времени через `WHERE ts >= ?`.

## Как расширить

- Добавить отслеживание конкретных jetton-адресов (мемкоинов), а не только
  роутеров — тогда алерты будут точечными "монета X купили на $Y".
- Хранить историю сделок в Postgres вместо SQLite — удобно при росте объёма
  данных и для построения дашбордов.
- Обернуть несколько ботов (volume alerts, new coins, whale buys) в единый
  сервис с общей конфигурацией.
- Добавить веб-дашборд (например, на Streamlit) поверх данных из `db.py`.
